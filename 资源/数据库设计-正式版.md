# 软件杯数据库设计（正式版）

> 架构讨论修订：2026/05/18  
> 本版已合入 Coze 智能体运行配置、Redis Stream 异步任务追踪、社区异步审核、代码题 AI 评阅与 Flyway V1 初始化口径调整。

## 一、设计目标与存储分层

本系统面向“高校个性化学习资源体系 + 多智能体协同学习系统”场景，数据库设计目标如下：

1. 支撑 Spark 端核心闭环：`用户画像 -> 个性化资源生成 -> 学习路径规划 -> 学习执行 -> 答题评测 -> 动态反馈`
2. 支撑 Manage 端核心闭环：`知识库上传 -> 文档解析 -> 切片向量化 -> RAG 检索 -> 争议工单 -> 知识修正`
3. 满足多智能体协同、RAG 检索增强、异步任务调度、进度跟踪、内容审核与可追溯审计需求
4. 结构遵循企业开发规范，减少冗余字段与弱关联设计，保证可直接落地开发

### 1.1 存储职责拆分

#### MySQL

负责结构化业务数据存储：

1. 用户与账号
2. 动态学习画像
3. 课程 / 知识点 / 学习资源
4. 学习计划、学习进度、答题记录、评测报告
5. 智能体会话元数据、异步任务、工单、后台权限
6. 知识库文档、切片、文件资产、解析策略

#### Redis

负责高频读写、短时状态与异步协同：

1. 登录态与黑名单
2. 智能体短期会话上下文
3. 学习时长、打卡、排行榜
4. 资源生成进度、向量化进度
5. 分布式锁、任务队列、热点缓存

#### Chroma

负责向量检索与 RAG 底座：

1. 官方权威知识库切片向量
2. 用户个人笔记 / 错题本向量
3. 视频字幕 / 多模态内容向量
4. 面向事实核查、资源生成、个性化推荐与智能答疑的检索能力

## 三、MySQL 关系型数据库设计

## 3.1 统一设计规范

### 3.1.1 命名规范

1. 表名统一使用小写下划线命名法，如 `sys_user`、`user_profile`
2. 主键统一命名为 `id`
3. 外键字段统一命名为 `xxx_id`
4. 枚举类字段统一命名为 `xxx_status`、`xxx_type`

### 3.1.2 公共字段规范

核心业务表统一包含以下公共字段：

- `id` bigint 主键
- `created_at` datetime 创建时间
- `updated_at` datetime 更新时间
- `is_deleted` tinyint 逻辑删除标记

说明：

1. 纯关系表可不强制保留 `id`，采用联合主键即可
2. 字典表可不强制 `is_deleted`，但建议保留 `status`
3. 流水表可按场景保留 `created_at`，不强制 `updated_at`

### 3.1.3 索引规范

1. 所有外键字段必须建普通索引
2. 高频查询字段建立联合索引
3. 唯一业务标识建立唯一索引
4. 大字段不建立索引

### 3.1.4 约束规范

1. 核心主链路建议落外键约束
2. 高并发表与历史流水表可在应用层控约束，但必须保留索引与注释
3. 所有状态字段需有明确取值说明

### 3.1.5 审稿精修原则

本版在正式版基础上进一步按“数据库审稿”标准做以下修正：

1. 删除可由上级实体唯一推导出的冗余字段，避免数据不一致
2. 保留少量统计类冗余字段，但明确其为“缓存/汇总字段”，以事实表为准
3. 补齐课程、知识点、资源、练习、工单、权限等主链路外键
4. 对日志类与高频流水类表，仅在必要处保留外键，避免过度约束
5. 让 RAG 元数据链路在 MySQL 中可回溯到 `课程 -> 知识点 -> 文档 -> 切片 -> 向量`

## 3.3 核心实体关系

### 3.3.1 核心主链路

1. `sys_user` 1:1 `user_profile`
2. `sys_user` 1:N `study_plan`
3. `study_plan` 1:N `study_task`
4. `course` 1:N `knowledge_node`
5. `knowledge_node` 1:N `learning_resource`
6. `knowledge_node` 1:N `question_bank`
7. `sys_user` N:N `knowledge_node`，通过 `user_knowledge_progress`
8. `sys_user` 1:N `user_question_record`
9. `sys_user` 1:N `evaluation_report`

### 3.3.2 RAG 主链路

1. `sys_oss_file` 1:1 / 1:N `kb_document`
2. `kb_parse_strategy` 1:N `kb_document`
3. `kb_document` 1:N `kb_chunk_record`
4. `kb_chunk_record` 1:1 `Chroma Vector Point`

### 3.3.3 管理后台链路

1. `sys_admin` N:N `sys_role`
2. `sys_role` N:N `sys_permission`
3. `feedback_ticket` N:1 `sys_user`
4. `feedback_ticket` N:1 `sys_admin`
5. `sys_operation_log` N:1 `sys_admin`

### 3.3.4 LangGraph 记忆链路

1. `sys_user` 1:N `agent_graph_thread`
2. `agent_graph_thread` 1:N `agent_graph_checkpoint`
3. `sys_user` 1:N `agent_memory`
4. `agent_graph_thread` 1:N `agent_memory`
5. `agent_memory` 按 `memory_scope` 分为：
   `public` 公共记忆
   `user_private` 用户私有长期记忆
   `agent_private` Agent 私有长期记忆
   `session_short` 会话级短期记忆
6. `agent_memory_summary` 用于滚动摘要，支撑长会话压缩与多轮续聊

## 3.5 扩展模块建议

以下模块建议作为二期补充，不影响比赛主链路交付，但数据库设计阶段仍应先列出表结构与约束方向，避免后期主表无法扩展。

### 3.5.1 博客与社区

建议至少保留以下表：

1. `blog_post`
2. `blog_comment`
3. `blog_like`
4. `blog_favorite`
5. `blog_view_log`

### 3.5.2 IM 即时通讯

不建议继续使用单表 `sys_message` 混合承载私聊、群聊、系统通知。

建议拆分为：

1. `im_conversation`
2. `im_message`
3. `im_message_read`
4. `sys_chat_group`
5. `sys_group_member`
6. `user_friend`

### 3.5.3 商城与周挑战

建议保留以下表：

1. `sys_achievement_badge`
2. `user_achievement`
3. `mall_item`
4. `user_asset`
5. `weekly_challenge`
6. `challenge_submission`

## 四、Redis 设计

## 4.1 命名规范

统一格式：

`业务域:子域:对象标识[:附加维度]`

示例：

`auth:user:token:{token}`

### 4.1.1 命名规则

1. 小写英文 + 冒号分隔
2. 避免同一个业务场景多种命名风格并存
3. 队列、锁、缓存、统计分别使用不同前缀

---

## 4.2 核心 Key 设计

| Key | 类型 | TTL | 说明 |
|---|---|---:|---|
| `auth:user:token:{token}` | String | 7天 | 存储 `user_id/session_id`，用户登录态 |
| `auth:admin:token:{token}` | String | 12小时 | 管理员登录态 |
| `auth:user:banned:{uid}` | String | 封禁期 | 用户封禁标记 |
| `profile:user:{uid}` | Hash | 12小时 | 用户画像缓存 |
| `agent:context:{uid}:{agent}` | List | 2小时 | 智能体最近多轮上下文 |
| `agent:stream:{session_id}` | Stream | 10分钟 | 流式输出缓冲 |
| `lg:thread:active:{uid}:{graph}` | String | 24小时 | 当前活跃 LangGraph 线程ID |
| `lg:checkpoint:latest:{thread_id}` | String | 24小时 | 最近 checkpoint 标识 |
| `memory:short:{thread_id}` | List | 2小时 | 短期记忆窗口 |
| `memory:summary:rolling:{thread_id}` | String | 24小时 | 当前滚动摘要缓存 |
| `memory:agent:private:{uid}:{agent}` | Hash | 24小时 | Agent 私有热点记忆缓存 |
| `memory:user:private:{uid}` | Hash | 24小时 | 用户私有热点记忆缓存 |
| `study:duration:daily:{uid}:{yyyyMMdd}` | String | 7天 | 当日学习时长累计 |
| `study:streak:{uid}` | String | 长期 | 连续学习天数 |
| `task:progress:{task_id}` | Hash | 2小时 | 资源生成进度、状态、消息 |
| `lock:generate:{uid}` | String | 10秒 | 资源生成防抖锁 |
| `lock:kb:parse:{doc_id}` | String | 30分钟 | 文档解析锁 |
| `queue:kb:parse` | Stream | 长期 | 文档解析队列 |
| `queue:kb:embed` | Stream | 长期 | 向量化任务队列 |
| `kb:progress:{doc_id}` | Hash | 24小时 | 解析与向量化进度 |
| `resource:detail:{resource_id}` | Hash | 12小时 | 学习资源详情缓存 |
| `rank:blog:daily` | ZSet | 24小时 | 博客热榜 |
| `rank:study:weekly` | ZSet | 7天 | 周学习排行榜 |
| `tts:cache:{md5}` | String | 30天 | TTS 音频缓存 |
| `oss:presigned:{file_id}` | String | 55分钟 | 文件签名 URL 缓存 |

---

## 4.3 Redis 使用说明

### 4.3.1 登录态

1. Token 本身不存敏感信息
2. Redis 仅存映射与过期时间
3. 强制下线通过删除 `auth:user:token:{token}` 实现

### 4.3.2 智能体上下文

1. 使用 `List` 存最近 N 条对话
2. 服务端需 `LTRIM` 保留最近 20 条
3. 长期历史仍以 MySQL 为准

### 4.3.3 LangGraph 分层记忆

推荐采用以下分层：

1. 短期记忆：
   Redis `memory:short:{thread_id}` + MySQL `agent_chat_message`
2. 滚动摘要：
   Redis `memory:summary:rolling:{thread_id}` + MySQL `agent_memory_summary`
3. 长期公共记忆：
   MySQL `agent_memory`，`memory_scope=public`
4. 用户私有长期记忆：
   MySQL `agent_memory`，`memory_scope=user_private`
5. Agent 私有长期记忆：
   MySQL `agent_memory`，`memory_scope=agent_private`
6. Graph 状态持久化：
   MySQL `agent_graph_checkpoint`

### 4.3.4 队列

正式设计中统一使用 `Stream`，不再写成 `List/Stream` 二选一，避免实现歧义。

### 4.3.5 缓存一致性

1. MySQL 更新后主动删缓存
2. TTL 增加随机值，防止雪崩
3. 对高频对象可加布隆过滤器，防止缓存穿透

## 5.1 Collection 设计

| Collection | 用途 | 向量维度 | 数据来源 | 主要场景 |
|---|---|---:|---|---|
| `edu_ground_truth` | 官方权威知识库 | 768 | 教材、课件、论文、课程文档 | 事实核查、资源生成、答疑引用 |
| `spark_personal_notes` | 用户个人笔记库 | 768 | 错题本、笔记、问答总结 | 个性化复习、查漏补缺 |
| `video_transcripts` | 视频字幕库 | 768 | 视频 ASR 转写文本 | 视频内容检索、秒级定位 |
| `multimodal_assets` | 多模态图文资源 | 768 | 截图、图解、思维导图 | 图文检索、辅助答疑 |

## 5.2 向量元数据设计

每条向量写入时应至少带以下 metadata：

```json
{
  "document_id": 1001,
  "chunk_id": 90001,
  "collection_name": "edu_ground_truth",
  "domain": "数据结构",
  "course_id": 11,
  "node_id": 203,
  "doc_type": "official_textbook",
  "source_file_id": 3001,
  "source_file_name": "data_structure_chapter6.pdf",
  "difficulty_level": "medium",
  "embedding_model": "text-embedding-v1",
  "embedding_version": "v1",
  "document_version": 2,
  "created_at": 1715481234
}
```

说明：

1. `document_id` 必须存在，便于按文档删向量
2. `chunk_id` 必须存在，便于定位与热更新
3. `node_id`、`course_id` 便于按知识点过滤
4. `embedding_model`、`embedding_version` 便于重嵌入治理

---

## 5.3 切片规则

### 5.3.1 文本教材

1. `chunk_size = 800 ~ 1200 tokens`
2. `chunk_overlap = 100 ~ 200 tokens`
3. 按自然段、标题层级优先切分

### 5.3.2 代码案例

1. 以函数、类、模块为主切片
2. 不强拆语义完整的代码块
3. Metadata 中补充 `language`、`module_name`

### 5.3.3 视频字幕

1. 按固定时间窗口或语义段切片
2. Metadata 增加 `start_time`、`end_time`

---

## 5.4 检索匹配策略

### 5.4.1 事实核查

1. 对生成内容提取关键问题
2. 生成查询向量
3. 在 `edu_ground_truth` 中召回 Top-K
4. 结合元数据过滤课程、知识点、版本

### 5.4.2 个性化推荐

1. 根据用户画像确定课程、难度、薄弱点
2. 检索 `edu_ground_truth + spark_personal_notes`
3. 按知识点优先级与难度进行重排

### 5.4.3 智能答疑

1. 优先检索当前课程、当前知识点
2. 不足时再检索跨章节相近知识
3. 输出时引用来源文档与切片摘要

---

## 5.5 Chroma 与 MySQL 联动

### 5.5.1 入库流程

1. 文件上传到对象存储，写入 `sys_oss_file`
2. 创建 `kb_document` 记录，状态 `pending`
3. 解析任务入 Redis `queue:kb:parse`
4. 切片后写入 `kb_chunk_record`
5. 向量化任务入 Redis `queue:kb:embed`
6. 向量写入 Chroma，并回写 `chroma_point_id`
7. 文档状态改为 `success`

### 5.5.2 删除流程

1. 逻辑删除 `kb_document` 与 `kb_chunk_record`
2. 按 metadata 中的 `document_id` 删除 Chroma 向量
3. 写 `sys_operation_log`

### 5.5.3 重解析流程

1. 删除旧向量
2. 归档或逻辑删除旧切片
3. 更新 `kb_document.document_version`
4. 重新入队解析与向量化

### 5.5.4 热更新流程

1. 教研员修改某个切片文本
2. 重新生成该切片向量
3. 更新对应 `chunk_id` 的 Chroma 记录
4. 无需重跑整本文档

## 六、LangGraph 记忆架构设计

## 6.1 记忆分层

### 6.1.1 短期记忆

用于当前线程窗口内的即时推理上下文：

1. 当前轮次消息
2. 最近 N 轮对话
3. 当前任务状态
4. 当前节点输入输出

存储建议：

1. Redis：高频读写
2. MySQL：消息落盘

### 6.1.2 滚动摘要

用于长会话压缩和减少 token 消耗：

1. 达到消息阈值后触发摘要
2. 用摘要替代早期详细上下文
3. 摘要可按线程 / Agent / 会话维度维护

存储建议：

1. Redis：当前摘要缓存
2. MySQL：摘要持久化

### 6.1.3 长期公共记忆

所有 Agent 可共享访问：

1. 用户目标
2. 用户长期偏好
3. 稳定学习风格
4. 长期薄弱点
5. 已验证的重要事实

存储位置：

`agent_memory.memory_scope = public`

### 6.1.4 用户私有长期记忆

仅属于用户本人，不一定对所有 Agent 全开放：

1. 个性化私人学习习惯
2. 隐私性目标
3. 敏感偏好配置
4. 个体连续学习轨迹特征

存储位置：

`agent_memory.memory_scope = user_private`

### 6.1.5 Agent 私有长期记忆

仅供某个 Agent 使用：

1. Sage 专属启发策略
2. Strict 专属任务调度经验
3. Coach 专属评测偏好与错题画像
4. Ava 专属鼓励话术偏好

存储位置：

`agent_memory.memory_scope = agent_private`

## 6.2 LangGraph 状态持久化建议

如果你后续采用 LangGraph，多 Agent 编排建议这样对应：

1. `agent_graph_thread`
   对应 LangGraph thread / run thread
2. `agent_graph_checkpoint`
   对应 checkpoint / graph state snapshot
3. `agent_chat_session`
   对应用户可见会话
4. `agent_chat_message`

5. `agent_memory`

6. `agent_memory_summary`

## 6.3 建议触发规则

### 6.3.1 短期转摘要

1. 单线程消息数超过 20~30 条
2. 或累计 token 超过阈值
3. 触发滚动摘要生成并压缩旧上下文

### 6.3.2 摘要转长期记忆

1. 当摘要中出现稳定偏好、长期目标、重复弱点时
2. 将其抽取为结构化长期记忆写入 `agent_memory`

### 6.3.3 评测反哺记忆

1. `evaluation_report` 生成后
2. 将知识薄弱点、解题习惯、错误模式写入长期记忆

## 八、架构讨论修订记录

本次根据 `arch-demo/arch.md` 的架构讨论结果，对数据库相关文件做了以下合入：

1. **新增智能体运行配置表**：增加 `sys_agent_config`，用于保存 Coze Bot / Workflow 的非敏感路由配置；`sys_agent_prompt` 保持提示词与模型参数配置职责。
2. **补齐 Coze 会话映射字段**：`agent_chat_session` 增加 `agent_config_id`、`external_conversation_id`、`external_section_id`；`agent_chat_message` 增加 `external_message_id`、`raw_response`。
3. **扩展异步任务追踪字段**：`async_generation_task` 增加 `task_type`、`external_run_id`、`retry_count`、`max_retry`、`started_at`、`finished_at`，用于 Redis Stream 任务消费、Coze Workflow 运行状态和重试追踪。
4. **调整社区审核状态**：`blog_post.post_status` 从 `1发布 / 0下架` 扩展为 `0草稿 1已发布 2待审核 3风险待复核 4审核拦截 5管理员下架`；`blog_comment` 新增 `comment_status` 并增加审核状态索引。
5. **明确代码题 MVP 策略**：代码题暂不新增沙箱相关表，继续复用 `question_bank.question_type = code` 与 `user_question_record.judge_mode = ai` 支撑代码文本提交和 AI 评阅。
6. **调整队列职责**：异步任务队列直接采用老版本 Redis Stream 设计，核心队列为 `queue:kb:parse` 与 `queue:kb:embed`；Redis 同时保留登录态、短期上下文、进度缓存、流式缓冲和分布式锁。
7. **调整 Flyway V1 口径**：Flyway `V1__init_hope_sparks_schema.sql` 应基于本次讨论后的 `hope_sparks.sql` 完整结构生成，而不是原始未修订版本再叠加 V2-V5 补丁。

## 九、结论

本版数据库设计相较初稿，完成了以下优化：

1. 明确了核心范围与二期扩展范围
2. 统一了公共字段、命名、索引、约束规范
3. 补齐了用户会话、课程、计划、资源版本、权限、工单、会话、RAG 治理字段

5. 使 `MySQL + Redis Stream + Chroma + MinIO` 的存储与任务结构更适合直接进入后端开发

如果后续进入实现阶段，建议优先开发以下主链路：

1. 用户登录与画像构建
2. 课程知识树与资源中心

4. 题库答题与评测报告
5. 知识库上传、切片、向量化与检索
6. 工单审核与内容修正闭环
